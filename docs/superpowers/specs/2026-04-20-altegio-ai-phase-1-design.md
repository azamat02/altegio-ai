# Altegio AI — Phase 1 Design Spec

**Date:** 2026-04-20
**Status:** Approved for implementation planning
**Scope:** Phase 1 of MVP — Telegram morning report bot

---

## 1. Context

Altegio AI is an analytics SaaS built on top of the Altegio API for beauty salons and clinics in KZ/UA/CIS. The product surfaces actionable signals (cancellation loss, staff performance, client churn) that Altegio's native analytics underserves. Partner API access is verified, tested against real data from location `198823` (anonymized as "Салон №1, Алматы").

The MVP was narrowed from five variants (A-E) to a phased combo of:
- **A** — Telegram morning report (daily hook)
- **B** — Loss-of-revenue report (wow moment)
- **C** — Staff performance dashboard as Telegram Mini App (deep engagement)

This document covers **Phase 1: Telegram morning report**. Phases 2 and 3 will get their own specs once Phase 1 ships.

## 2. Decisions (recorded upfront)

| Decision | Chosen | Rationale |
|---------|--------|-----------|
| Tenancy model | Multi-tenant architecture with **closed onboarding** | Avoid rewrite pain later; self-service is premature before feedback loop exists |
| Feature sequencing | **Phased by infrastructure** — Telegram bot first | The bot builds ~60% of shared infra (sync, metrics, scheduler); subsequent phases layer on top |
| AI narrative | **Hybrid** — template metrics + single Claude-generated insight | Numbers stay deterministic; AI adds perceived intelligence without risking hallucinated figures |
| Stack | NestJS + TypeScript, PostgreSQL + TypeORM, Redis + BullMQ, Telegraf, Claude API, Docker Compose on VPS | Already aligned with founder's stack experience; no K8s complexity in Phase 1 |
| Scope of Phase 1 | Morning Telegram report only — no web UI, no billing, no self-service signup | Ship in 3-4 weeks; customer development with real product in hand |

## 3. Phase 1 Scope

### 3.1 Functional end-state

Every morning at the tenant's configured `report_time` (default 09:00 in the salon's timezone, e.g. Asia/Almaty), the salon owner receives a single Telegram message from `@AltegioAIBot`. Delivery completes within ~10 seconds of the scheduled minute (compute + Claude call + Telegram send). The message contains:

1. **Yesterday's summary** — revenue vs 7-day baseline, visits (completed/cancelled), average check
2. **Top-3 staff** by yesterday's revenue
3. **Attention section** — 0-3 bullets driven by rules: cancellation spike, struggling staff, low today-load (section hidden if no rules trigger)
4. **Today's outlook** — booked records, occupancy percent, empty slots
5. **Main insight** — 1-2 sentence commentary generated by Claude from the computed metrics (section hidden on failure)

Example message body is referenced in the brainstorming transcript; the template renderer is the canonical source of truth for format.

### 3.2 In scope

| Component | Responsibility |
|----------|----------------|
| Periodic Altegio sync | Every 6 hours, pull delta of records/clients and snapshots of staff/services for each enabled tenant into Postgres |
| Metrics computation | At 08:55 local time, compute yesterday/baseline/today figures from facts tables |
| Template renderer | Deterministic string assembly; hides empty sections |
| AI insight service | Claude call with guardrails, timeout, validation, fallback to null |
| Delivery | At 09:00 local time, send via Telegraf to the tenant's configured chat ID |
| Admin CLI | Commands `add-salon`, `link-telegram`, `trigger-report`, `trigger-sync` |
| Observability | `/health` endpoint, structured logs, Sentry for errors, `sync_jobs` + `report_deliveries` + `ai_insight_logs` audit tables |

### 3.3 Explicitly out of scope (Phase 1)

- Web UI (onboarding, dashboard) → Phase 3 TMA
- Loss-of-revenue report → Phase 2
- Self-service salon registration
- Billing / subscription
- User-facing settings (time, TZ, enabled sections) — configured via DB/CLI
- Altegio webhooks (polling only)
- Historical backfill beyond 60 days

### 3.4 Acceptance criteria

1. BrowUp tenant receives a correctly-formatted message at 09:00 on three consecutive days without manual intervention
2. A new tenant can be onboarded end-to-end in under 5 minutes via `pnpm cli add-salon <token> <location_id> <telegram_chat_id>`
3. Claude API failure degrades gracefully — the message still ships without the "Main insight" section
4. A failure in one tenant's pipeline does not prevent another tenant's report from being sent

## 4. Data Architecture

### 4.1 Three-layer model

Raw payloads, normalized facts, and pre-computed aggregates are separated because each serves a different purpose: raw enables replay/debugging without re-hitting Altegio, facts power queries with indexes, aggregates make report generation fast and deterministic.

```
Layer 1: RAW      — JSONB blobs from Altegio (altegio_raw_records, …)
          ↓ parser
Layer 2: FACTS    — normalized rows (records, clients, staff, services)
          ↓ aggregator
Layer 3: AGGREGATES — per-tenant-per-day views (daily_metrics, staff_daily, …)
```

### 4.2 Core tables (Phase 1 subset)

```sql
-- Multi-tenancy root
tenants (
  id uuid PK,
  salon_name text,
  location_id bigint,
  altegio_token text,            -- stored as pgp_sym_encrypt(token, app_key)
  chain_id bigint,
  timezone text,                 -- e.g. 'Asia/Almaty'
  telegram_chat_id bigint,
  report_enabled bool,
  report_time time,              -- default '09:00'
  working_hours_per_day int,     -- default 10; used for occupancy heuristic in Phase 1
  created_at, updated_at
)

-- Layer 1: RAW
altegio_raw_records (
  id bigserial PK,
  tenant_id uuid FK,
  altegio_record_id bigint,
  payload jsonb,
  fetched_at timestamptz,
  UNIQUE (tenant_id, altegio_record_id)
)
-- analogous: altegio_raw_clients, altegio_raw_staff, altegio_raw_services

-- Layer 2: FACTS
records (
  id bigserial PK,
  tenant_id uuid FK,
  altegio_record_id bigint,
  staff_id bigint,
  client_id bigint,
  datetime timestamptz,
  seance_length int,
  cost numeric(12,2),
  attendance smallint,        -- 0=pending, 1=arrived, -1=no-show, 2=confirmed
  paid_full smallint,
  is_online bool,
  deleted bool,
  created_at timestamptz,
  INDEX (tenant_id, datetime),
  INDEX (tenant_id, staff_id, datetime)
)
-- analogous: staff, clients, services

-- Layer 3: AGGREGATES
daily_metrics (
  tenant_id uuid, date date,
  revenue_total numeric(14,2),
  visits_completed int,
  visits_cancelled int,
  avg_check numeric(12,2),
  occupancy_pct numeric(5,2),
  PRIMARY KEY (tenant_id, date)
)
staff_daily (
  tenant_id uuid, staff_id bigint, date date,
  revenue numeric, visits int, cancelled int, avg_check numeric,
  PRIMARY KEY (tenant_id, staff_id, date)
)

-- Infra
sync_jobs (id, tenant_id, status, started_at, finished_at, records_fetched, error)
report_deliveries (tenant_id, date, message_id, sent_at, status, error,
                   PRIMARY KEY (tenant_id, date))
ai_insight_logs (tenant_id, date, prompt_hash, response, ms, status, created_at)
```

### 4.3 Sync strategy

- **Cadence**: BullMQ cron, every 6 hours per tenant (`report_enabled=true`)
- **Delta window**: `since = last_sync - 1 hour` (overlap catches late-updated records). UPSERT by `(tenant_id, altegio_record_id)` is idempotent.
- **Rate limiting**: Altegio caps at 200 req/min. Enforce client-side via `p-limit` + exponential backoff. At 10 tenants × ~75 delta records per 6h window, we are well under the limit.
- **Snapshots**: `staff` and `services` (small, slow-changing) are pulled as full snapshots on each sync, not deltas.
- **Backfill**: On new tenant onboarding, a separate `backfill` queue pulls 60 days of history slowly. Deeper backfill is manual.
- **Soft delete**: When Altegio marks a record as deleted, the fact row gets `deleted=true`. Reports filter `WHERE deleted = false`.

### 4.4 Security

- `altegio_token` encrypted at rest via `pgp_sym_encrypt` using `APP_ENCRYPTION_KEY` (32-byte secret in VPS env, not in git)
- Database dumps and logs therefore cannot leak partner tokens
- All service methods take `tenantId` explicitly; every query filters on `tenant_id` — enforced by ESLint rule in Phase 1, migrated to TypeORM global scope later

### 4.5 Idempotency

- Sync: UPSERT on unique (tenant, altegio_id) keys
- Report delivery: `report_deliveries` primary key is `(tenant_id, date)` — prevents duplicate morning messages
- Aggregates: recomputed from facts for any changed date on each sync

## 5. Backend Module Structure

### 5.1 NestJS module layout (apps/api)

```
src/
├── modules/
│   ├── tenants/           # CRUD tenants, token encryption helpers
│   ├── altegio/           # HTTP client; only place that knows Altegio auth/rate-limit/retry
│   ├── sync/              # Orchestrates raw → facts → aggregates
│   │   ├── parsers/       # raw → facts
│   │   ├── aggregator.service.ts
│   │   └── sync.processor.ts  # BullMQ consumer
│   ├── metrics/           # Pure DB reads; returns DailyReportData
│   ├── reports/           # Business logic: metrics → template → ai → deliver
│   │   ├── template.renderer.ts
│   │   ├── ai-insight.service.ts
│   │   └── reports.processor.ts  # BullMQ consumer
│   ├── telegram/          # Telegraf wrapper
│   ├── scheduler/         # Cron triggers → enqueues jobs
│   └── health/            # /health
├── queues/                # BullMQ setup: sync, backfill, reports
├── db/migrations/         # TypeORM SQL migrations
├── config/                # ConfigModule + zod env validation
└── main.ts
```

### 5.2 Module boundaries

| Module | Public interface | Rationale |
|--------|-----------------|-----------|
| `altegio` | `client.fetchRecords(tenant, since)` | Single integration point — if Altegio V2 becomes available, only this changes |
| `sync` | `syncTenant(tenantId)` | Encapsulates raw→facts→aggregates; knows nothing about reports |
| `metrics` | `getDailyReportData(tenantId, date) → DailyReportData` | Pure Postgres reads; reusable by Phase 3 TMA |
| `reports` | `generateAndDeliver(tenantId, date)` | Product logic of "morning message" — composes metrics + template + ai + telegram |
| `ai-insight` | `getInsight(reportData) → string \| null` | Isolated so (a) failures return null and reports knows to fall back, (b) unit-testable with fixtures |
| `telegram` | `sendReport(chatId, text) → messageId` | Only place talking to Telegram API |
| `scheduler` | `@Cron` handlers → enqueue jobs | Separated from processors so tests can run without firing crons |

### 5.3 Report orchestration flow

```
[wide cron every minute]
  scheduler.tick()
    for each tenant where report_enabled = true:
      if current time in tenant.timezone equals tenant.report_time (default 09:00):
        queue.add('generate-report', { tenantId, date })
              ↓
[reports.processor]
  reports.service.generateAndDeliver(tenantId, date):
    ├── metrics.getDailyReportData(tenantId, date)       ← Postgres, ~50ms
    ├── template.render(data) → baseText
    ├── ai-insight.getInsight(data) → insight | null
    │     └── Claude call, 3s timeout, try/catch → null on failure
    ├── final = baseText + (insight ? "\n\n💡 Main insight\n" + insight : "")
    ├── telegram.sendReport(tenant.telegram_chat_id, final) → messageId
    └── INSERT INTO report_deliveries (tenant_id, date, message_id, status='sent')
           ^ UNIQUE (tenant_id, date) prevents duplicate send
```

### 5.4 Runtime topology

Phase 1 runs as a single NestJS process that is simultaneously HTTP server, BullMQ producer, and BullMQ worker. A future `WORKER_ENABLED=true` env flag will allow splitting producer/worker when scale demands — not needed in Phase 1.

### 5.5 DI and testability

`altegio`, `telegram`, and `ai-insight` are injected via interfaces (`IAltegioClient`, `ITelegramSender`, `IAiProvider`). Unit tests mock these so `reports.service` logic is verified without network calls.

### 5.6 CLI

`apps/cli` bootstraps the same `AppModule` with HTTP/cron disabled, exposes commands: `add-salon`, `link-telegram`, `trigger-report <tenant>`, `trigger-sync <tenant>`. Commands call services directly — no logic duplication.

## 6. Morning Report Details

### 6.1 Metrics computed for the report

| Metric | Source | Method |
|--------|--------|--------|
| Yesterday revenue, visits, cancels, avg check | `daily_metrics` | Single row lookup |
| 7-day baseline (avg revenue, avg visits, avg cancel rate) | `daily_metrics` | AVG over `[date-8 .. date-2]` |
| Top-3 staff by yesterday revenue | `staff_daily` + `staff` | JOIN, ORDER BY revenue DESC LIMIT 3 |
| Struggling staff | `staff_daily` | revenue_yesterday < avg_7d * 0.6 AND below_avg_previous_day |
| Today's booked count | `records` | COUNT where `datetime::date = today AND deleted = false` |
| Today's occupancy % | `records` + tenants.working_hours_per_day | `SUM(seance_length_for_today_seconds) / (active_staff_count × tenants.working_hours_per_day × 3600) × 100`. Working hours per Phase 1 are a tenant-level constant (default 10) until `staff_schedule` is synced in a later phase |
| Empty slot hints | `records` | Hours with no booking vs a fixed 10:00-20:00 window (simple heuristic for Phase 1) |
| Cancellation loss | `records` | SUM(cost) WHERE attendance = -1 AND yesterday |
| Cancellation clusters | `records` JOIN `staff` | Group by staff × hour, order DESC, top 3 |

All queries cached in Redis for 30 minutes under key `metrics:daily:{tenantId}:{date}` to de-duplicate debug/retry calls.

### 6.2 Template renderer

Plain TypeScript function (no Handlebars) in `template.renderer.ts`, ~80 lines. Shape:

```
☀ Доброе утро! {salon_name} · {weekday}, {date}

📊 Вчера
• Выручка: {rev} ₸ {delta_vs_baseline}
• Визитов: {visits} (пришли) / {cancelled} (отменили, {pct}%)
• Средний чек: {avg_check} ₸

🏆 Топ-3 мастера
{ranked list}

⚠ Требует внимания
{attention bullets — 0 to 3}

📅 Сегодня
• {booked} записей, загрузка {pct}%
• Пустые слоты: {hours} (if any)
```

`deltaVsBaseline()` returns `(+12% к среднему за неделю)` / `(−7% к среднему за неделю)` / `""` if abs delta < 3%.

`buildAttention()` applies rules in priority order (max 3 bullets):
1. If `cancel_rate > baseline.cancel_rate * 1.3` → "Рост отмен: {n}, потеря ~{X}K ₸"
2. For each struggling staff (max 2) → "{name} — {n}-й день подряд ниже среднего"
3. If `today_load < 40%` → "Низкая загрузка сегодня"

If no rule triggers, the "Требует внимания" section is omitted entirely.

### 6.3 AI insight

**Contract:** input is `DailyReportData` (computed numbers only, never raw records). Output is 1-2 Russian sentences, max 280 characters.

**Prompt:**
```
System: Ты — аналитик салона красоты. Тебе дают факты за вчера.
Твоя задача — найти ОДИН самый важный инсайт в 1-2 предложениях
на русском. Правила:
- Говори только о переданных фактах. НЕ ПРИДУМЫВАЙ ЦИФРЫ.
- Не повторяй цифры, которые уже есть в отчёте.
- Если есть аномалия — называй её и возможную причину.
- Если всё нормально — коротко подтверди тренд.
- Максимум 280 символов. Без эмодзи.

User: {JSON of DailyReportData}
```

**Guardrails (in `ai-insight.service.ts`):**

1. **Timeout** — 3 seconds; on timeout return `null`
2. **Response validation:**
   - length ≤ 280 chars
   - all numbers in response must be a subset of numbers in input (regex extract, compare)
   - strip markdown / emoji
3. **Rate limit** — client-side Bottleneck at 5 req/s against Anthropic
4. **Cache** — `insight:{tenantId}:{date}` in Redis so retries don't re-bill
5. **Audit log** — every call persisted to `ai_insight_logs` with `{input_hash, output, ms, status}`

### 6.4 Error handling

| Failure point | Behavior |
|--------------|----------|
| Altegio API unreachable during sync | Retry with exponential backoff × 3 → on fail, `sync_jobs.status = 'failed'`. Report still ships using previous data. If `last_sync > 12h ago`, template prepends "⚠ данные не обновлены с {timestamp}" |
| Postgres down | App crashes → Docker restart → if still failing after 5 min, Sentry alert to founder's Telegram (not client's) |
| Claude timeout / error | Insight = null → report ships without "Main insight" section. WARN logged |
| Telegram API error | Retry × 2 → on fail, `report_deliveries.status = 'failed'`, alert. No automatic next-day retry |
| No data for yesterday (empty day) | Template gracefully shows "визитов не было" — no crash |
| Owner blocked the bot (403) | Set `tenant.report_enabled = false`, log, founder investigates manually |

### 7.6 Testing strategy

| Level | Location | Tooling | Coverage |
|-------|----------|---------|----------|
| **Unit** | `*.spec.ts` next to code | Jest | `template.renderer` (snapshot from fixture), `buildAttention` rules matrix, `ai-insight.validate` (accept/reject responses), `token-cipher` round-trip |
| **Integration** | `apps/api/test/` | Jest + testcontainers | `sync.service` end-to-end (mock Altegio client, real Postgres/Redis), `metrics.service` (seed facts → assert metrics), `reports.service` (mock AI/TG → assert message composition) |
| **E2E (manual)** | Dev / staging | CLI `trigger-report <tenant>` against real BrowUp token | Full wire run |
| **Fixtures** | `test/fixtures/` | Sanitized JSON snapshots of Altegio API responses | Shared across unit + integration, zero network calls in CI |

**Mandatory tests for Phase 1:**

1. `template.renderer` produces expected text from happy-path fixture
2. `template.renderer` correctly hides sections when data is empty (edge: no visits, no staff)
3. `buildAttention` against rule matrix (normal day / cancel spike / struggling staff)
4. `ai-insight.validate` rejects responses containing fabricated numbers
5. `sync.parser` idempotent: double sync yields no duplicates (UPSERT works)
6. `aggregator` produces correct `daily_metrics` rows from synthetic records
7. `metrics.service` matches hand-calculated values on a known fixture
8. `reports.service` fallback: when AI returns null, message still composes and sends
9. `token-cipher` round-trip encryption
10. `report_deliveries` UNIQUE constraint catches duplicate send

Coverage is not a KPI. The goal: if `template.renderer.ts` or a SQL statement in `metrics.service.ts` is touched, tests catch regressions.

## 9. Non-goals / deferred decisions

- **Altegio V2 API** — not functional with current partner token; revisit when V2 opens
- **Webhooks** — polling is sufficient for daily cadence; webhooks considered in Phase 2/3 if we want intra-day alerts
- **WhatsApp/SMS** — deferred; Telegram covers target audience for validation
- **Multi-language** — Russian only for Phase 1
- **User-facing settings UI** — config via DB/CLI until Phase 3 TMA adds a settings screen
